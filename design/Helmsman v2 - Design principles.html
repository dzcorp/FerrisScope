Design principles for Helmsman

Helmsman is a control plane for a system that is already complex, often broken, and never quiet. Our job is to make it feel boring. These are the axioms, principles, and rules we use to keep the surface calm as we add features — and the checklist we run before shipping any one of them.

01 Axioms

Three claims about our user that everything else inherits from. If a feature contradicts one of these, the feature is wrong, not the axiom.

A1

The operator is on call

They opened Helmsman because something is wrong, or might be. Latency, ambiguity, and unread modal copy translate directly into incident minutes. Optimise for the 2 a.m. case, not the demo.

A2

The operator already knows Kubernetes

They speak in kubectl, YAML, and resource kinds. We translate between their mental model and a faster surface — we don't replace it. Never invent terminology that maps to an existing K8s noun.

A3

One wrong click can cost a weekend

Destructive actions, scope changes, and "apply" are not the same as filtering or selection. They get friction proportional to blast radius, every time, no exceptions for power users.

02 Principles

Eight directional rules that resolve disagreements during design. Cite them by number in reviews.

P1

State first, chrome second

// the truth of the cluster is the page

Headers, rails, and toolbars exist to frame data, not compete with it. When a screen feels busy, take chrome away before you take data away. The pod table is the product.

Do

Cluster bar shows live CPU/MEM gauges and namespace context, then gets out of the way.

Don't

Add marketing-style "welcome" panels, tip carousels, or onboarding nudges to the cluster view.

P2

One canonical action per moment

// the right next click should be obvious

Every screen has one primary action; the rest are secondary, ghost, or hidden in ⋯. If two buttons compete for attention, one of them is in the wrong screen.

P3

Speed is a design property

// the keyboard is the fastest pointing device

Every common path has a keyboard route. ⌘K finds anything; ⌘` opens a terminal; Esc always backs out. Mouse is for discovery; keyboard is for the third visit.

P4

Friction scales with blast radius

// dangerous actions look dangerous

Read-only is one click. Selection-aware bulk action is two. Destructive (Delete, Drain, Cordon, Apply to prod) is a typed confirmation with the resource name. Make the cost of being wrong visible before the action commits.

P5

Status is structural, not decorative

// colour means the same thing everywhere

Green = healthy / running. Amber = transient / pending. Red = failing / crashed. Grey = unknown / paused. Accent (teal) is for UI state, never for status. Don't reuse status colours for selection, links, or branding.

P6

Show the cluster you're standing in

// context is never implicit

Cluster name + namespace scope must be visible on every screen that reads or writes. Terminal prompts, dock chips, and confirmations all repeat it. Operators with three production clusters open trust scaffolding more than they trust themselves.

P7

Density without noise

// every glyph has to earn its row

We are an information-rich tool. That doesn't license decoration. No emoji, no gradients on data, no rounded badge for every cell. If a number can replace a chart, use the number. If a row can replace a card, use the row.

P8

Persist what the operator built

// the workspace remembers

Open terminals, draft YAML, namespace filters, sort orders, pinned rail — all survive a refresh, a tab switch, and a cluster change where it makes sense. State the operator constructed is more valuable than state we computed.

03 Concrete rules

The principles above, made enforceable. These are checked in design review.

R-01
Live data refreshes silently. No spinners on poll, no skeletons on re-fetch. Use a 1-px progress bar in the cluster bar if you must signal activity.
tables · charts
R-02
Tables sort, filter, and select with the keyboard. Up/Down moves the row cursor; Space selects; Enter opens detail; / focuses filter; Esc clears in that order.
tables
R-03
Bulk actions appear in a docked bar, never as a row of buttons in a header. The bar slides up only when ≥1 item is selected and disappears on clear.
tables
R-04
Destructive verbs are never the default action of a button group. Delete is always the trailing item in a menu, never the rightmost button. Confirmations require typing the resource name.
all
R-05
YAML is the source of truth. Every form-driven edit must show the underlying YAML diff before Apply. No hidden mutations.
forms · apply
R-06
Numbers use tabular figures. font-variant-numeric: tabular-nums on every column that holds counts, percentages, durations, or memory.
type
R-07
Identifiers are monospace. Pod names, image tags, hashes, IPs, namespaces in chips. Sentence text is sans. Don't mix the two in a single token.
type
R-08
Hit targets are ≥ 28 px. Icon-only buttons are 28×28; menu items 32 high; dense table rows 30 minimum. The mouse is precise; the operator is tired.
controls
R-09
Modals are the exception. Prefer a side panel for detail, a dock tab for editing, an inline confirmation for one-off actions. Modals only for namespace pickers, destructive confirmations, and the command palette.
layout
R-10
Empty states explain what would fill them. "No pods in this namespace" plus the kubectl command that would have created some. Never just an icon and "Nothing here yet."
empty
R-11
Errors quote the server. If the API said Error from server (Forbidden), we render that string verbatim and add our own one-line interpretation underneath. We don't paraphrase k8s into prose.
errors
R-12
The accent colour is teal, and only teal. It marks active tab, focus ring, primary button, and selection. Reserve it. If everything is accented, nothing is.
colour
R-13
Esc always backs out one layer. Order: command palette → settings → namespace modal → detail panel → bulk selection. Never close the whole app.
keyboard
R-14
Animations are under 200 ms and only on user action. Slide-in dock, fade-in modal, drawer expansion. Never animate data; never animate on poll.
motion

05 Anti-patterns

Things we have specifically decided not to do, with reasons. Every one of these has been pitched and rejected.

"AI-suggested fix" buttons in error states

The operator is deciding whether to drain a node holding a stateful service. A confident-sounding suggestion from a system that doesn't share their context is worse than nothing. We surface logs, events, and the kubectl command — they decide.

Dashboards as the landing page

Generic time-series charts on a homepage train the operator to ignore them. We land on the cluster picker because that is the choice they came to make. Detail charts live inside resource pages, where they're scoped to something specific.

Inline drag-and-drop reordering of resources

Order in Kubernetes comes from labels, selectors, and priority classes — not from a UI affordance. Letting someone drag pods around builds a mental model the cluster doesn't honour.

Wizards for resource creation

"Step 3 of 7" is the wrong shape for a Deployment. The dock's YAML scratchpad with templates and dry-run is the right shape: full surface, real format, instant feedback. Wizards train people in our model, not Kubernetes'.

Toasts that announce success after a write

A toast that says "Pod restarted" is information the table can already show. Update the row, animate the status pill back to green, and trust the operator to read. Toasts are reserved for things that happened off-screen.

"Pro tips" and onboarding tooltips on hover

The operator has done this 400 times. We document keyboard shortcuts in Settings → Shortcuts and let the surface be quiet. First-run guidance is fine; perpetual coach-marks aren't.

06 Feature review checklist

Run this list before shipping any new feature. If you skip an item, write down why in the design ticket.

Scope & framing

  • The feature has a one-sentence description in operator vocabulary (no internal codenames, no metaphors).
  • It maps to an existing K8s concept, or it explicitly extends one with a name we will commit to.
  • The "do nothing" option was considered and rejected for a stated reason.

Surface

  • It lives in an existing region (rail, header, dock, detail panel) — or a new region was added with sign-off.
  • It introduces ≤ 1 new primary button per screen.
  • Empty, loading, and error states are designed — not just the populated state.
  • Cluster + namespace context is visible from the feature without scrolling.

Interaction

  • Every action has a keyboard route (or a stated reason it can't).
  • Esc behaviour is defined and consistent with the existing stack.
  • Selection-aware bulk actions appear in the dock bar, not in the header.
  • Destructive paths require typed confirmation; the resource name appears in the prompt.

Style

  • Uses only tokens from §04 — no new colours, no new font sizes, no new radii.
  • Identifiers are monospace; numbers are tabular; status uses the documented colours.
  • No emoji, no gradients on data, no decorative iconography.

Data & honesty

  • If the feature writes to the cluster, the YAML diff is visible before Apply.
  • Server errors are quoted verbatim with our interpretation underneath.
  • Live values refresh silently; no spinner on poll.
  • Anything the operator constructs (filters, drafts, terminal sessions) survives a refresh.

Accessibility & performance

  • Hit targets ≥ 28 px; focus rings visible on every interactive element.
  • Colour is never the only signal — pairs with shape, label, or position.
  • Tested with the dark theme; tested with system reduced-motion; tested at 1280-wide.
  • Initial render < 100 ms perceived; interaction response < 50 ms.
